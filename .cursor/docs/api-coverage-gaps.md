# API Coverage Gaps

This document records strong user demand that UnifAPI Skills should acknowledge but cannot fully satisfy with the current UnifAPI public API surface. It does not treat generic host-agent abilities as UnifAPI API gaps.

Current API surface checked on 2026-05-27 from `https://api.unifapi.com/openapi.json`: Instagram, LinkedIn, Reddit, Threads, TikTok, X, and YouTube. These APIs support a strong public-social GTM and monitoring portfolio, but they do not yet cover several Okara-like and Clay-like workflow primitives.

Most target runtimes for these skills, including ChatGPT, Codex, and Claude Code, already provide web search, URL fetch/browser fetch, shell access, JS/TS/Python execution, and local file editing. Skills may use those capabilities directly. A missing UnifAPI endpoint is only a product/API gap when the workflow needs hosted repeatability, bulk throughput, normalized source-specific schemas, saved state, notifications, private OAuth connectors, write/publish actions, compliance metadata, or billing/rate-limit control.

Several P0/P1 gap categories now have skills. Those skills should remain discoverable and useful even before every API is ready. When host-agent capabilities are enough for one-off execution, the skill should run the workflow and label the source path. When a workflow needs productized scale, persistence, private connectors, or write actions, the skill must name the API backfill below instead of claiming UnifAPI already provides it.

## Dependency Model

| Dependency tier | Skill behavior | Examples | API implication |
| --- | --- | --- | --- |
| Agent-native now | Use the host agent's search, fetch/browser, shell, code execution, and local file editing directly. | Public web research, one-off page extraction, small CSV/table transformation, known-URL SEO checks, HN/public-page monitoring, repo SEO patches. | Do not block the skill. Document API backfill only for productizing scale, repeatability, storage, alerts, or normalized schemas. |
| Current UnifAPI | Use MCP OAuth and current public social/professional APIs. | X, Reddit, LinkedIn, YouTube, TikTok, Instagram, Threads monitoring and creator/GTM workflows. | Publish as executable UnifAPI-backed skills. |
| API backfill required | Use a planned API only when the host runtime cannot safely or repeatedly provide the capability. | Email verification, CRM writes, private SaaS connectors, autonomous publishing, ad account actions, saved monitors, on-chain indexers, bulk enrichment. | Keep as API-backfill or partial until endpoints/connectors exist. |

## Product Implications

- The first Workflow Portfolio should be positioned as public-social GTM intelligence plus agent-native web research, SEO/page checks, SEO content, growth analytics exports, backlink/mention checks, table enrichment, account scoring, creator discovery, social listening, competitor monitoring, and web3 social monitoring.
- Skills must not promise full Clay platform replacement until domain/company enrichment, email finding, CRM export/write APIs, persistent tables, private connectors, and scheduling exist. One-off CSV/table enrichment can still be agent-native.
- Skills must not promise full Okara product replacement until autonomous channel execution, scheduled monitors, publishing, durable brand memory, and private workspace connectors exist. One-off web, HN, SEO, and content workflows can still be agent-native.
- Web3 skills are useful today for narrative and community monitoring, but on-chain and market-data APIs are required before UnifAPI can claim full web3 intelligence.

## Recommended API Roadmap Buckets

1. **Web and SEO productization**: SERP, keyword, hosted URL scrape, rendered extraction, sitemap crawl history, page metadata, answer-engine visibility.
2. **GTM enrichment foundation**: domain-to-company, company enrichment, social-handle discovery, email finder/verifier, firmographics, tech stack, persistent table runtime.
3. **Monitor infrastructure**: saved queries, schedules, diffs, thresholds, notification destinations, event history.
4. **Private connectors**: CRM, email, calendar, Slack, Notion, support tools, analytics tools, with clear OAuth scope and audit metadata.
5. **Web3 data foundation**: wallet/contract/token/NFT/governance/market data, then social-plus-on-chain correlation.

## How Skills Should Reference Gaps

When a user asks for unsupported coverage, the agent should:

1. State what the host agent can do directly and what the current UnifAPI API can support.
2. Use agent-native search/fetch/shell/code for one-off public workflows when that will produce real evidence.
3. Name the missing API category only when the user needs scale, persistence, private data, write actions, normalized schemas, or productized repeatability.
4. Add the gap to future planning if it is a repeated or high-value request.
